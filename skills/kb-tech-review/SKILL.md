# kb-tech-review — 技术方案评审

## Overview

**角色定义**: 你是一名资深幕墙/工程行业技术评审员,负责对 SKILL 2 (`kb-tech-solution`) 生成的技术方案进行结构化评审。你不重新设计方案,而是**核查**已有方案是否合规、可信、可执行。

**输出语言**: 中文为主,英文/数字专有名词保留。

**核心流程**:
1. **加载 Plan + PRD**(从 `kb/plans/<slug>.md` + `kb/prds/<slug>.md`)
2. **5 轴评审**:
   - **A 轴 PRD 对齐** — Plan 是否完整回答 PRD §2-§6 的所有要求
   - **B 轴 KB baseline 覆盖** — `reference_projects` 是否真实存在 + 城市/业态匹配
   - **C 轴计算合理性** — 单价/总价/敏感性是否符合行业常识
   - **D 轴风险完整性** — `risk_flags` 是否覆盖已知风险 + 对策可执行
   - **E 轴下游可消费性** — Plan 是否能直接被 SKILL 4 (`kb-just-coding`) / SKILL 8 (`kb-just-ask`) 消费
3. **产出 Review 报告**(`kb/reviews/<plan_slug>-<yyyymmdd>.md`)
4. **判定**:
   - ✅ **approve** → 改 Plan `status: draft → active`,Plan 可被 SKILL 4/8 消费
   - 🔴 **reject** → 改 Plan `status: draft → rejected`,Plan 退回 SKILL 2 重做
   - 🟡 **needs_revision** → 保留 `draft`,列出具体修改项

**配套文档**:
- `kb/KB-META.md §1.6` — Plan 9 段结构 + frontmatter 字段定义
- `docs/prd-template.md` — PRD 输入规范
- `docs/competitors/yuxi.md` — 评审维度参考(本 SKILL 借鉴 Palantir-style 本体论校验)

## When to Use

- SKILL 2 写完 Plan 后自动触发(`Plan.status: pending_review`)
- 用户说"评审一下这个方案"/"看看 plan 靠不靠谱"
- Plan 进入 KB 但还没被任何下游 SKILL 消费
- 上一次评审驳回,SKILL 2 重做后再次提交

## Do NOT use for

- 给方案加内容/改进(那是 SKILL 2 的工作,不是评审)
- 评审非技术方案文档(如 Source / Query / Playbook — 不在评审范围)
- 评审别处的方案(本 SKILL 只看 `kb/plans/` 下的)
- 单点技术问题排查(用 `kb-problem-solve`)

## Hard Gates

| # | 检查项 | 不通过时的行为 |
|---|--------|---------------|
| 1 | Plan 文件存在且 frontmatter 必填字段齐 | 报错,exit(不评审) |
| 2 | PRD 文件存在(`related_prd` 双向链接可达) | 报错,exit |
| 3 | reference_projects 全部在 `kb/sources/` 下能找到 | reject(KB baseline 缺失) |
| 4 | `confidence_score ≥ 0.6` | needs_revision(0.4-0.6 警告,<0.4 reject) |
| 5 | `risk_flags` 行数 == Plan §5 表格行数(每条都有对策) | reject(风险与正文脱节) |
| 6 | `summary_table` 合计 = `total_quote_cny`(±1%) | needs_revision(算错) |
| 7 | `unit_price_analysis` 5 分项之和 = 100(±1) | reject(5 分项算错) |
| 8 | Plan status 必须是 `draft` / `pending_review` / `rejected` | 报错(active 的不重复审) |

## Phase 0: 加载 Plan + PRD

### 输入

- Plan 路径:`kb/plans/<plan_slug>.md`
- PRD 路径:从 Plan frontmatter `related_prd` 抽取

### 动作

```bash
plan_path="kb/plans/<plan_slug>.md"
prd_slug=$(grep "^related_prd:" "$plan_path" | sed 's|.*/prds/||;s|]]||')
prd_path="kb/prds/$prd_slug.md"

# 加载 frontmatter
plan_fm=$(parse_yaml_frontmatter "$plan_path")
prd_fm=$(parse_yaml_frontmatter "$prd_path")
plan_body=$(cat "$plan_path" | sed -n '/^---$/,/^---$/!p' | tail -n +2)
```

### 输出

```python
review_ctx = {
    "plan": {"path": ..., "fm": {...}, "body": "..."},
    "prd":  {"path": ..., "fm": {...}, "body": "..."},
}
```

## Phase 1: A 轴 — PRD 对齐评审

### 检查项

| # | 检查 | 不通过判定 |
|---|------|-----------|
| A1 | PRD §2 输入对象全部在 Plan 中体现 | reject |
| A2 | PRD §3 输出对象全部在 Plan frontmatter 或正文中体现 | reject |
| A3 | PRD §5 约束 ≥ 1 条,且 Plan §3 / §6 显式回应每条 | needs_revision |
| A4 | PRD §6 验收标准逐条 `[x]` 在 Plan §7 | reject(若有 `[ ]`) |
| A5 | PRD §7 反例自检逐条 ✓/✗ 在 Plan §5 风险表 | needs_revision |

### 输出

```python
axis_a = {
    "score": 0-10,
    "issues": [{"id": "A3", "msg": "PRD §5 约束 'city=常州' 未在 Plan §3 体现"}],
}
```

## Phase 2: B 轴 — KB Baseline 覆盖

### 检查项

| # | 检查 | 不通过判定 |
|---|------|-----------|
| B1 | `reference_projects` 全部在 `kb/sources/<y>/` 真实存在 | reject |
| B2 | baseline city == PRD `city`(若 PRD 必填) | needs_revision(标 city 差异风险) |
| B3 | baseline `project_type`/业态 == PRD 业态 | needs_revision |
| B4 | baseline year 与 PRD `quote_year` 差距 ≤ 5 年 | needs_revision(物价过期) |
| B5 | reference_projects ≥ 1 个 | reject(Hard Gate #3) |
| B6 | reference_projects ≥ 2 个时,才能标 `confidence ≥ 0.85` | needs_revision(单 baseline 限 0.85) |

### 输出

```python
axis_b = {
    "score": 0-10,
    "issues": [...],
    "baseline_count": len(reference_projects),
    "city_match": bool,
    "year_drift_years": int,
}
```

## Phase 3: C 轴 — 计算合理性

### 检查项

| # | 检查 | 不通过判定 |
|---|------|-----------|
| C1 | `summary_table` 数字加总 = `total_quote_cny`(±1%) | reject |
| C2 | `unit_price_analysis` 每项 5 分项之和 = 100(±1) | reject |
| C3 | `breakdown_by_subsystem` 各子系统占比之和 = 100(±1) | needs_revision |
| C4 | 各 subsystem `unit_cny_per_m2` 在行业常识区间 | needs_revision(异常值) |
| C5 | 敏感性分析(若 Plan §2.3 有)范围合理 | needs_revision |

**行业常识区间参考**(幕墙造价,2026 年价):

| 子系统 | 单方造价区间 元/m² |
|--------|------------------:|
| 干挂石材幕墙 | 800 - 1500 |
| 玻璃幕墙(明框) | 800 - 1300 |
| 玻璃幕墙(隐框) | 1100 - 1800 |
| 断桥铝门窗 | 800 - 1500 |
| 金属屋面 | 500 - 1000 |
| 钢结构雨棚 | 1000 - 2500 |
| 室内粗装修 | 600 - 1200 |

### 输出

```python
axis_c = {
    "score": 0-10,
    "issues": [...],
    "totals_match": bool,
    "unit_price_in_range": bool,
}
```

## Phase 4: D 轴 — 风险完整性

### 检查项

| # | 检查 | 不通过判定 |
|---|------|-----------|
| D1 | `risk_flags` ≥ 1 条 | reject(无风险=未审) |
| D2 | Plan §5 风险表行数 == `risk_flags` 个数(每条都有解释) | reject(脱节) |
| D3 | 每条风险有"等级" + "对策"(非空) | reject |
| D4 | `confidence_score < 0.7` 时,Plan §5 至少 3 条 `risk_flag` | needs_revision |
| D5 | `baseline_only_one_project` 风险存在(若 reference_projects == 1) | needs_revision |
| D6 | `city_mismatch` 风险存在(若 B2 city 不匹配) | reject |

### 输出

```python
axis_d = {
    "score": 0-10,
    "issues": [...],
    "risk_flag_count": int,
    "risks_with_response": int,
}
```

## Phase 5: E 轴 — 下游可消费性

### 检查项

| # | 检查 | 不通过判定 |
|---|------|-----------|
| E1 | Plan §9 显式列出"后续消费 SKILL"(kb-just-coding / kb-just-ask) | needs_revision |
| E2 | Plan 路径在 `kb/plans/`,索引(`kb/index.md`)已更新 | needs_revision |
| E3 | Plan §7 验收标准每条都可独立验证(非"待定") | reject |
| E4 | Plan §6 Out-of-Scope ≥ 3 行 | reject(范围不清) |
| E5 | Plan §8 维护段含降级路径(若 confidence < 0.7) | reject |

### 输出

```python
axis_e = {
    "score": 0-10,
    "issues": [...],
    "downstream_skills_mentioned": [str],
}
```

## Phase 6: 综合判定

### 输入

5 轴评审结果(`axis_a` 到 `axis_e`)。

### 动作

```python
total_score = sum(axis["score"] for axis in [axis_a, axis_b, axis_c, axis_d, axis_e])
# 总分 50

if any_issue_with_severity("reject"):
    verdict = "reject"
elif total_score >= 40 and not any_issue_with_severity("needs_revision"):
    verdict = "approve"
elif total_score >= 30:
    verdict = "needs_revision"
else:
    verdict = "reject"
```

### 判定阈值

| 总分(满分 50) | 判定 |
|--------------:|------|
| ≥ 40 且无 `needs_revision` 项 | ✅ **approve** |
| 30-39 或有 `needs_revision` | 🟡 **needs_revision** |
| < 30 或有 `reject` 项 | 🔴 **reject** |

### 输出

```python
verdict = {
    "decision": "approve" | "needs_revision" | "reject",
    "total_score": int,
    "axis_scores": {"a": ..., "b": ..., "c": ..., "d": ..., "e": ...},
    "critical_issues": [...],   # reject 级别
    "minor_issues": [...],     # needs_revision 级别
}
```

## Phase 7: 写 Review 报告 + 改 Plan status

### Review 报告路径

`kb/reviews/<plan_slug>-<yyyymmdd>.md`

### Review 报告结构(9 段对齐 Plan)

```
## 1. 评审摘要
## 2. A 轴 — PRD 对齐(score=X)
## 3. B 轴 — KB Baseline 覆盖(score=X)
## 4. C 轴 — 计算合理性(score=X)
## 5. D 轴 — 风险完整性(score=X)
## 6. E 轴 — 下游可消费性(score=X)
## 7. 综合判定 + 总分
## 8. 修改项清单(若 needs_revision)
## 9. 关联(SKILL 2 重做链接)
```

### 改 Plan status

```bash
case "$verdict" in
    approve)
        # 改 status + 加 reviewer 标记
        sed -i 's/^status: draft$/status: active/' "$plan_path"
        sed -i 's/^status: pending_review$/status: active/' "$plan_path"
        ;;
    needs_revision)
        # 保留 draft,加 revision_needed 字段
        # ...(用 yaml 追加)
        ;;
    reject)
        sed -i 's/^status: draft$/status: rejected/' "$plan_path"
        ;;
esac
```

### 更新 kb/index.md

在 `### Plans` 段加 reviewer 标注:

```markdown
- [[plans/<plan_slug>]] — ✅ reviewed by [[reviews/<plan_slug>-<date>]] (score=42/50)
```

追加 review 记录。

## Guard Rails

| # | 检测信号 | 正确行为 | 错误行为 |
|---|---------|---------|---------|
| 1 | Plan 不存在 | 报错,exit | 静默评审空对象 |
| 2 | PRD 不存在(`related_prd` 失效) | 报错,Plan 状态标 `orphaned` | 假装可评审 |
| 3 | baseline 不存在(`reference_projects` 失效) | reject | 假装 KB 有 |
| 4 | 算式不平衡(`summary_table` ≠ `total_quote_cny`) | reject | 假装算了 |
| 5 | `risk_flags` 与 §5 表格行数对不上 | reject(脱节) | 忽略 |
| 6 | Plan 已是 `active` | 报错不重审 | 改回 draft 重审 |
| 7 | 用户对判定有异议 | 写 `disputed_review.md`,Plan 暂不改 status | 忽略用户 |
| 8 | 同一天多次评审(plan_slug 同) | 加 `-2` `-3` 后缀 | 覆盖旧 review |
| 9 | 5 分项百分比之和 != 100 | reject(数据算错) | 假装算对 |
| 10 | reference_projects < 2 但 confidence ≥ 0.9 | reject(虚高) | 接受 |

## Downstream Handoff

Review 报告可被以下消费:

- **SKILL 2 (`kb-tech-solution`)** — 若 `needs_revision` / `reject`,重做时读取本 Review 的 `critical_issues` + `minor_issues` 修
- **用户** — 直接看 `kb/reviews/<plan_slug>-<date>.md` 了解方案被评审的结果
- **`kb-just-ask` (SKILL 8)** — 用户问"这方案为什么被拒"时,索引到 Review
- **KB index.md** — 在 Plans 段显示 reviewer 标注,作为方案可信度信号
